How the Oracle GP Model Works
==============================
A complete explanation — architecture, math, and code — written to be understood
and re-explained to anyone from a collaborator to a thesis committee.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
PART 1: THE PROBLEM
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

We want to predict: how strongly does Drug D bind to Protein A?

Binding affinity is measured as pIC50 (or pKi, pKd) — the negative log of the
inhibitory concentration. Higher = tighter binding. Values typically range 5–9.

The catch: we are in a STREAMING COLD-START setting.

Proteins arrive one at a time. When Protein A arrives, we have never seen it
before — no binding data for A exists yet. But we *have* seen Drug D bind to
other proteins, and we've seen other drugs bind to A's structural neighbors.
That accumulated history is our signal.

This is the virtual screening problem: given a new, unstudied protein target,
rank hundreds of candidate drugs by predicted affinity before we run any assays.
Getting the ranking right early saves enormous wet-lab cost.

The prequential protocol (test-then-train):
  1. Protein A arrives.
  2. Predict affinities for all query drugs for A — BLIND, using only history.
  3. Evaluate prediction quality (Concordance Index, Enrichment Factor).
  4. Labels are revealed. Train on them. Add to history.
  5. Next protein arrives. Repeat.

Every prediction is genuinely zero-shot at the time it's made. The protocol is
honest — no label leakage. "Episode" = one protein's screening event.

PART 2: THE DATA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The dataset is stored as a PyTorch Geometric HeteroData object with:

  Node types:
    "protein"   — UniProt proteins. Features: ESM-2 language model embeddings 
                  (2816 dims) concatenated with structure-derived features (CATHedral).
                  These capture sequence-level and evolutionary information.
    "drug"      — ChEMBL small molecules. Features: GRAPHEIN created graphs, MolCLR to derive the embeddings
    "go"        — Gene Ontology terms. Nodes in the protein-GO bipartite graph.

  Edge types:
    ("protein", "binds_pic50", "drug")   — pIC50 binding measurements
    ("protein", "binds_pki", "drug")     — pKi binding measurements
    ("protein", "binds_pkd", "drug")     — pKd binding measurements
    ("protein", "similar_to", "protein") — protein-protein sequence similarity
    ("protein", "annotated_by", "go")    — protein-GO term annotations
    ("drug", "similar_to", "drug")       — drug-drug Tanimoto similarity

Because pIC50, pKi, and pKd measure the same underlying thing (binding
affinity in different assay formats), they are merged into a single edge type
"binds_activity" using a max-reduce:

    merge_activity_edges(data, reduce="amax")

For any protein-drug pair that appears in multiple metrics, take the maximum
(strongest evidence). This gives one canonical affinity per pair.

In addition to the HeteroData features, we have 3D structural representations
constructed using Graphein (a Python library for biological graph construction):

  - Protein graphs: built from PDB structures. Each residue becomes a node
    (11 features: amino acid type, secondary structure, solvent accessibility,
    etc.) and edges connect residues within a spatial distance cutoff (13
    features: Cα distance, bond type, dihedral angles). Stored in
    protein_graphs.zip.
  - Drug graphs: built from SMILES strings. Each atom becomes a node (20
    features: atom type, formal charge, hybridization, aromaticity, ring
    membership, etc.) and edges are bonds (5 features: bond type, stereo
    configuration, conjugation, ring membership). Originally stored in 2410
    tar.gz archives; packed into drug_graphs_packed.pt for fast loading.

These structural graphs are used to train the MolGraphPrior (see Part 5).

PART 3: PROTEIN AND DRUG REPRESENTATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

For a protein A and drug D, we need fixed-length vector representations.

PROTEIN FEATURES:
  The HeteroData node features data["protein"].x are typically ESM-2 embeddings
  (a protein language model trained on evolutionary sequences). ESM-2 embeds a
  protein sequence into a 2816-dim vector that implicitly encodes:
    - Structural motifs (alpha helices, beta sheets)
    - Binding pocket characteristics
    - Evolutionary conservation
  Two proteins with similar ESM cosine similarity tend to have similar binding
  profiles, especially for drugs targeting conserved motifs.

DRUG FEATURES:
  data["drug"].x contains precomputed drug feature vectors encoding:
    - Morgan fingerprints (circular fingerprints of radius 2): binary vectors
      indicating presence of substructure fragments
    - Physicochemical descriptors (MW, logP, TPSA, etc.)
  Two drugs with high cosine similarity in this space tend to have similar
  binding profiles across proteins (they hit the same pocket types).

3D STRUCTURAL FEATURES (for MolGraphPrior):
  The flat feature vectors capture chemical identity but not 3D structure.
  The molecular graphs (GINE encoder, see Part 5) read the actual 3D geometry.
  These graphs are constructed using Graphein — see Part 2 for full feature
  details. In brief:
    - Protein graphs: residue nodes (11-dim) + distance-cutoff edges (13-dim).
      Captures how residues are connected in 3D, active-site geometry,
      angles and distances.
    - Drug graphs: atom nodes (20-dim) + bond edges (5-dim). Captures bond
      topology, atom types, ring systems, and 3D conformation.

These two representations are complementary. The flat features are fast and
good for retrieval and similarity scoring. The graph encoder is slower but
more expressive for function prediction.

PART 4: THE STREAMING PROTOCOL — HOW EPISODES WORK
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

build_multiplex_stream(data) creates the episode sequence.

It finds all proteins with ≥ 10 known binding measurements (so CI is
statistically meaningful), shuffles them randomly (so episode difficulty is
uniform, not sorted by how well-studied the protein is), and creates a
ProteinEpisode for each.

A ProteinEpisode contains:
  - protein_idx: integer index of this protein
  - edges: [2, n_drugs] — all (protein, drug) pairs with known affinities
  - labels: [n_drugs] — the pIC50/pKi/pKd values

The historical initialization (--historical-protein-frac):
  A fraction of proteins (e.g. 50%) are pre-loaded into the revealed history
  before the streaming starts. This simulates the practical scenario where we
  have prior assay data from related proteins in the same program. Cold episodes
  (early in the stream) still have no exact drug-protein history but can
  draw on the historical context.

For each episode:
  1. Context is built from the revealed history.
  2. All query drugs are scored simultaneously.
  3. CI and EF@5% are computed against the true labels.
  4. The episode's labels are added to the history (add_revealed).
  5. A gradient step updates the GP model using the TNP loss.

After every --checkpoint-every episodes, a snapshot is saved to disk.

PART 5: THE PRIOR — WHERE DOES THE BASELINE PREDICTION COME FROM?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Every GP needs a prior: a prediction before seeing any data. We have two options.

OPTION A: BindingEncoder (default, --gnn-prior none)
  A small MLP trained offline on all historical binding data:
      prior(A, D) = MLP(concat(protein_features[A], drug_features[D]))
  This learns a basic function-valued prior — "for proteins like A and drugs
  like D, affinity tends to be around X." It's trained with MSE loss on
  mean-centered targets and then frozen.

  The prior is used in two ways:
    1. As the GP prior mean μ₀ for cold-start predictions
    2. To compute residuals: residual(A, D) = label(A, D) - prior(A, D)
       The GP operates on residuals — it models the deviation from the prior,
       not the raw affinity. This centers the regression task.

OPTION B: MolGraphPrior (--gnn-prior mol)
  A more powerful prior that uses the actual 3D molecular graphs.

  Architecture (src/models/mol_graph_encoder.py):
    - ProteinGraphEncoder: GINE (Graph Isomorphism Network with Edge features)
      with 4 layers, 256 hidden dim. Reads the protein residue graph.
    - DrugGraphEncoder: GINE with 4 layers, 256 hidden dim. Reads the drug
      atom-bond graph.
    - BilinearScoringHead: projects both embeddings to rank-128 space and
      takes their dot product → scalar affinity prediction.

  GINE works like this:
    Each layer takes node features and edge features, and updates each node
    by aggregating messages from its neighbors:
        x_v^{l+1} = MLP(x_v^l + sum_{u ∈ N(v)} f(x_u^l, edge_uv))
    The edge features (distance, bond type, etc.) are projected to the same
    dimension as node features and added to the message. This means the
    graph convolution is aware of the specific type of connection between
    nodes — not just their existence. After all layers, global_mean_pool
    takes the average of all node embeddings to produce a single graph-level
    embedding.

  Pretraining (scripts/pretrain_mol_graph_prior.py):
    The MolGraphPrior is trained supervised on binding measurements from
    HISTORICAL proteins only — the same split used by the streaming experiment.
    This is critical to avoid leakage: if the GINE encoder were trained on
    streaming proteins' binding labels, those proteins would not truly be
    unseen at inference time.

    The pretrain script accepts three args that must match the streaming
    experiment:
      --historical-protein-frac (e.g. 0.5)  — fraction of stream used for
                                               pretraining
      --stream-seed (default 42)             — protein shuffle seed
      --stream-min-edges (default 15)        — minimum edges per protein

    The encoder replicates the same build_multiplex_stream + split logic,
    identifies the historical protein set, and filters training edges to
    those proteins only.

    After pretraining, protein and drug embeddings are precomputed for EVERY
    entity (not just historical ones) using the frozen encoder. This is clean:
    applying a frozen function to a graph is just feature extraction — no
    binding labels from streaming proteins were ever used in supervision. The
    encoder learned a mapping from graph structure to embedding space using
    only historical data; we then apply that mapping to all proteins including
    streaming ones.

    Loss = MSE on (affinity - global_mean_affinity). Uses AMP (mixed
    precision) for VRAM efficiency.

  These embeddings are saved to mol_prior_tables.pt. At inference, the prior
  is just a lookup + dot product — no graph forward pass needed.

  Why is the mol prior better than the BindingEncoder?
    The BindingEncoder only sees the protein's ESM vector and the drug's
    fingerprint. The MolGraphPrior reads the actual 3D structure — it can
    see which residues are in the binding pocket, how the drug fits spatially,
    whether there are steric clashes, etc. This gives a much more physically
    grounded prior.

PART 6: GAUSSIAN PROCESS REGRESSION — THE MATH
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

A Gaussian Process (GP) puts a probability distribution over functions. Instead
of assuming f is linear or quadratic, the GP says: "I don't know the shape,
but I believe nearby inputs have similar outputs" (measured by a kernel k).

Before seeing any data, our belief about the affinity f(A, D) is:
    f(A, D) ~ N(μ₀, σ₀²)
where μ₀ = prior(A, D) (from BindingEncoder or MolGraphPrior).

After seeing observations X = {(X₁,Y₁), (X₂,Y₂), ..., (Xₙ,Yₙ)} with
residual affinities r = y - μ₀, the GP posterior at (A, D) is:

    μ(A,D) = μ₀ + k(A,D; X) · [K(X,X) + σ²I]⁻¹ · r
    σ²(A,D) = k(A,D; A,D) - k(A,D; X) · [K(X,X) + σ²I]⁻¹ · k(X; A,D)

where:
  k(A,D; X)     = row vector of similarities between (A,D) and each context pair
  K(X,X)        = matrix of pairwise similarities among context pairs
  σ²I           = observation noise (regularization)
  r             = residual affinities (deviation from prior)

Let's unpack each piece:

  k(A,D; X):
    "How related is the query pair (A,D) to each observed pair (X_i, Y_i)?"
    High k((A,D), (X_i,Y_i)) means: "knowing the affinity at (X_i,Y_i) gives
    us a lot of information about the affinity at (A,D)."

    The key insight: this kernel should be CROSS-PAIR. It should measure
    similarity between the full observation (protein AND drug), not just protein
    to protein. Why? Because a context observation "(ProtB, DrugY, aff=7)" is
    informative about "(A, D)" only to the extent that:
      - A is similar to ProtB (same binding pocket region), AND
      - D is similar to DrugY (same chemical scaffold)
    If either similarity is near zero, the observation tells us nothing.

    The original code had a bug here: it encoded context as enc(X_i, D_query)
    instead of enc(X_i, Y_i). This is wrong — it completely ignored whether
    the observation drug Y_i is similar to D. The fix: cross-pair kernel.

  K(X,X):
    "How similar are the context observations to each other?"
    This matters for deduplication. If ProtB and ProtC are nearly identical,
    observing both is almost redundant — they carry the same signal. The matrix
    inversion [K(X,X) + σ²I]⁻¹ accounts for this and down-weights redundant
    observations.

  [K(X,X) + σ²I]⁻¹:
    The matrix inversion is the expensive step — O(n³) where n is the number
    of context observations. This is why we approximate it with attention.

  r = y - μ₀:
    We don't operate on raw affinities. The GP models the RESIDUAL after the
    prior prediction. If the prior already predicts 7.0 and the true value is
    7.5, the residual is 0.5. The GP only needs to explain the unexplained part.

PART 7: THE DEEP KERNEL — LEARNING THE SIMILARITY FUNCTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

A classical GP uses a hand-crafted kernel — RBF, Matérn, etc. We learn ours.

The DrugConditionalEncoder (src/models/gp_affinity.py):
    φ(protein, drug) → embedding of dimension d
    implemented as a 3-layer MLP with GELU activations and LayerNorm output.
    Input: concat(protein_features, drug_features)

The kernel is then:
    k((A,D), (X,Y)) = φ(A,D) · φ(X,Y) / sqrt(d)   [dot product]

This is DEEP KERNEL LEARNING. φ learns the relevant geometry:

  "Two proteins that differ only in regions irrelevant to D's binding pocket
   map close together; proteins that differ along D-relevant dimensions map
   far apart."

Why "drug-conditional"? Because binding pocket relevance is drug-specific.

Example: Aspirin (COX inhibitor) and a kinase inhibitor both bind proteins.
  - From Aspirin's perspective: COX-2 structure matters; kinase fold is irrelevant.
  - From the kinase inhibitor's perspective: ATP binding domain matters; COX fold
    is irrelevant.

Two proteins that look similar in ESM space might be very different from
Drug D's perspective (their pockets diverge exactly where D binds) and vice
versa. φ(A, D) projects A into the subspace that Drug D actually cares about.

The full cross-pair kernel is critical here. When we encode context observations
as φ(X_i, Y_i) where Y_i is the ACTUAL observation drug, the model learns:
  - "Is X_i's binding pocket in the region D-relevant space near A?"
  - "Is Y_i in the drug-space region near D?"
Both conditions must hold for the observation to be highly informative.

How does φ learn this geometry? Through the loss.

If the GP predicts affinity 7 for (A, D) because φ(A,D) ≈ φ(ProtB, DrugY)
and ProtB-DrugY has affinity 7, but the true affinity is 4, the gradient
pushes φ(A,D) away from φ(ProtB, DrugY) and toward pairs with affinity ~4.
Over thousands of episodes the embedding geometry self-organizes so that
nearby points have similar affinities.

THIS IS THE KERNEL. The embedding geometry IS the kernel. Training φ is
training the kernel function.

PART 8: HOW ATTENTION APPROXIMATES THE GP POSTERIOR
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Computing [K(X,X) + σ²I]⁻¹ exactly is O(n³) — too slow when n grows large.
Attention is a fast approximation.

Standard attention:
    output = softmax(Q · K^T / sqrt(d)) · V

where:
    Q (Query)  = what you're looking for
    K (Keys)   = labels on each piece of information
    V (Values) = the actual information content

The GP transfer step in our model:
    Q = φ(A, D)              → query pair embedding               [1, d]
    K_i = φ(X_i, Y_i)        → context pair embeddings            [n, d]
    V_i = r_i                → residual affinities (scalars)       [n, 1]

    transfer_logits = Q · K^T / sqrt(d)   [1, n]
    transfer_weights = softmax(transfer_logits) · ctx_mask
    transfer = sum_i(transfer_weights_i · r_i)    → scalar

The attention weights approximate k((A,D), X) · [K(X,X) + σ²I]⁻¹:
  - Q · K^T / sqrt(d) captures the raw kernel similarities k((A,D), (X_i,Y_i))
  - softmax normalization approximates the inversion [K + σ²I]⁻¹ under the
    assumption that context observations are reasonably diverse (not identical)

The result: transfer is a weighted average of residual affinities where the
weights measure "how informative is each context observation for this query?"

Final prediction:
    μ(A,D) = prior(A,D) + α · (transfer + correction)

where α is the context quality gate (see Part 9), and residuals stored in
context are (label - prior), so adding prior back gives absolute scale.

PART 9: THE CORRECTION HEAD AND CONTEXT GATE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The GP transfer is a LINEAR interpolation over scalar affinities. Real binding
landscapes have nonlinear structure — a protein might be 90% similar to ProtB
along most dimensions but have a point mutation in exactly the binding pocket
that completely flips affinity.

THE CORRECTION HEAD:
  A multi-layer cross-attention block (CrossAttentionLayer × n_layers) builds
  a rich attended representation of the context:
      x = Q_emb                                      [n_qry, 1, d]
      for each layer:
          x = cross_attn(query=x, context=K_emb)     [n_qry, 1, d]
      attended = LayerNorm(x).squeeze(1)             [n_qry, d]

  Unlike the GP transfer (single-head, over scalar values), this is multi-head
  (4 heads) attention over the rich d-dimensional context embeddings. Different
  heads can attend to different structural aspects of how Drug D's binding
  varies across protein space.

  The correction head then reads:
      output = MLP(concat(Q_emb, attended))          [n_qry, 2]
      mu_residual = output[:, 0]   → nonlinear correction to transfer
      log_sigma   = output[:, 1]   → predicted uncertainty

  Zero-initialization: the final linear layer of the MLP is initialized to
  all zeros. This means at training start, mu_residual = 0 and the model
  IS a pure GP transfer. Corrections are only learned where the data forces
  them. This is a principled way to regularize: "start simple, add complexity
  only as needed."

THE CONTEXT GATE (α):
  A key question: should we trust the GP transfer at all?

  If the context is good (exact drug matches, many observations), yes.
  If the context is weak (distant drug analogs from unrelated protein families,
  level-3 fallbacks), maybe not — the prior might be more reliable.

  A learned gating scalar α ∈ (0, 1):
      α = sigmoid(linear(attended))

  Bias initialized to -2, so sigmoid(-2) ≈ 0.12. The model starts near
  "ignore context, trust prior" and learns to open the gate as context
  proves reliable through the training stream.

  Final mean:
      mu_ctx = prior + α × (transfer + mu_residual)
      mu = mu_ctx  if has_context  else  prior

  When no context exists (fully masked), the model falls back to the prior
  exactly — no attention outputs are used at all.

  Predicted uncertainty (sigma):
      sigma = softplus(log_sigma) + ε
  softplus ensures sigma > 0. The +ε prevents numerical issues.
  The model calibrates sigma to reflect genuine uncertainty — when the GP
  transfer is confident (tight weights on a single high-quality context point),
  sigma should be small. When context is spread out or conflicting, sigma
  should be large.

PART 10: CONTEXT RETRIEVAL — THE PRODUCT KERNEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Before the GP can run, we need to select the K most informative context
observations from the growing history. This is the job of DrugFirstContextBuilder
(src/data/drug_first_context.py).

The history is stored as three flat lists:
  _pool_prot[i], _pool_drug[i], _pool_aff[i]
representing every revealed (protein, drug, residual_affinity) triple.

For each query (Protein A, Drug D), we score ALL pool entries by the
PRODUCT KERNEL:

    score(A, D; X_i, Y_i) = k_prot(A, X_i) × k_drug(D, Y_i)

where:
  k_prot(A, X_i) = (cosine_similarity(ESM(A), ESM(X_i)) + 1) / 2
                   ∈ [0, 1], normalized so 0 = opposite, 1 = identical

  k_drug(D, Y_i) = (cosine_similarity(drug_feat(D), drug_feat(Y_i)) + 1) / 2
                   ∈ [0, 1], same idea

The product kernel means BOTH protein and drug similarity must be high for
an observation to get a high score. If only one is high, the score is still
low. This is the correct behavior for the cross-pair kernel:
  - Exact (A, D) match → k_prot ≈ 1, k_drug = 1 → score ≈ 1 (top priority)
  - Drug analog (A, D') → k_prot ≈ 1, k_drug = drug_sim → score ≈ drug_sim
  - GO-similar protein (X, D) → k_prot = go_sim, k_drug = 1 → score = go_sim
  - Cross-transfer (X, D') → score = go_sim × drug_sim
  - Unrelated → both near 0 → score ≈ 0 (ignored)

The previous architecture had a hard-coded LEVEL HIERARCHY (Level 1: exact drug
history, Level 2: drug analogs, Level 3: GO-similar protein fallback). This was
an approximation to the product kernel. The product kernel replaces the hierarchy
entirely — the levels EMERGE naturally from the scores.

The GO-similar protein boost (role_boost):
  Proteins sharing GO terms with A have been observed to have Spearman ~0.747
  correlation with A's binding profile. This signal is important.

  Where does it come from? It comes from the PROTEIN-PROTEIN RELATIONSHIP, not
  from encoding GO term embeddings. Proteins share GO terms because they have
  related functions → they bind related drug classes → their binding profiles
  correlate. The GO term text itself is irrelevant; what matters is the
  relationship graph.

  (This is why we removed the GOEnricher from MolGraphPrior — encoding GO
  term vectors into protein embeddings was adding complexity without helping.
  The GP already captures the GO signal via the product-kernel boost on
  GO-similar proteins.)

  Implementation: pillar["role_binds_ei"] contains the edge index of proteins
  in the "role" layer (GO-annotated neighbors of A). For these proteins,
  k_prot is replaced with max(ESM_sim, role_boost). With role_boost=1.0,
  GO-similar proteins are treated as if they had full protein similarity,
  regardless of their ESM distance.

  Why max() instead of override? Because some GO-similar proteins may also
  be highly ESM-similar. We don't want to lower their score.

Large pool subsampling:
  When the history grows large (> max_pool = 4096 entries), we randomly
  subsample to keep the O(n) similarity computation tractable. Exact drug
  matches are always preserved (inserted separately, outside the subsample)
  so level-1 context is never lost.

We take the top K = 32 scored entries and pass them to GPAffinityModel as:
  ctx_proteins[i, :K]   — feature vectors of the K context proteins
  ctx_drugs[i, :K]      — feature vectors of the K observation drugs (Y_i, not D)
  ctx_affinities[i, :K] — residual affinities
  ctx_mask[i, :K]       — True for valid slots (padded with False)

PART 11: THE MULTIPLEX PILLAR STRUCTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The multiplex refers to multiple relational "layers" of information about the
same set of proteins:

  1. Binding layer: which drugs has this protein bound, and at what affinity?
  2. Sequence layer: which other proteins are sequence-similar?
  3. GO layer: which other proteins share GO annotations (functional similarity)?
  4. Structure layer: which proteins have similar 3D folds?

For each target protein A, a "pillar" is assembled from the multiplex graph —
a subgraph centered on A that contains:
  - Its neighborhood in each relational layer
  - The edges connecting those neighbors (role_binds_ei for GO-similar neighbors)
  - Features for all nodes in the pillar

The pillar is passed to build_context() to identify which history entries
come from GO-annotated neighbors (for the role_boost) and to provide
the contextual structure for the GP.

The MultiplexPillarSampler (src/data/multiplex_loader.py) handles construction
of the pillar for each target protein during the streaming loop.

PART 12: THE LOSS FUNCTION — TNP LOSS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Virtual screening is fundamentally a RANKING problem: "which drugs should we
test first?" Getting the top 5% right matters more than getting the middle
ranks right. Pure MSE doesn't reflect this.

We use a combined loss (src/training/tnp_loss.py):

1. Gaussian NLL (negative log likelihood):
       NLL = 0.5 * log(2π) + log(σ) + (y - μ)² / (2σ²)
   This jointly trains μ (the mean prediction) and σ (the uncertainty estimate).
   When σ is small and the prediction is wrong, the penalty is large. When σ
   is large (honest uncertainty), the penalty for being wrong is smaller.
   The model learns to be well-calibrated: confident predictions should be
   accurate, uncertain predictions should be less penalized.

2. ListNet ranking loss:
       target_probs = softmax(y / T)
       pred_log_probs = log_softmax(μ / T)
       ListNet = -sum(target_probs * pred_log_probs)
   This is a listwise ranking loss. Temperature T=2 is used because raw pIC50
   values (5–9) produce very peaked softmax distributions — without temperature
   scaling, almost all the gradient flows through the single highest-affinity
   drug, ignoring the rest of the ranking. T=2 spreads the gradient across
   the full distribution.

3. Lambda-MART pairwise loss:
       for all pairs (i,j) where y_i - y_j > 0.5:
           lambda_loss += softplus(-(μ_i - μ_j))
   This penalizes pairs where the model gets the relative order wrong by more
   than 0.5 pIC50 units. "Penalize inversions among clearly different drugs."
   The threshold 0.5 ignores near-ties (where the assay noise is larger than
   the difference anyway).

The three losses are combined with HOMOSCEDASTIC UNCERTAINTY WEIGHTING:
       L_total = Σ_k [ exp(-log_var_k) × L_k + 0.5 × log_var_k ]

Each log_var_k is a learnable parameter. The formula is derived from a
multi-task Gaussian likelihood:
  - If loss k is small and easy, log_var_k stays near 0 (equal weight)
  - If loss k is hard or noisy, log_var_k increases (downweights it)
  - The 0.5 * log_var term prevents trivially setting log_var → ∞

Initialized at log_var = 0 for all three → initial weights all 1.0.
Over training, the model learns to balance NLL, ListNet, and Lambda-MART
automatically, without manual tuning of loss weights.

PART 13: METRICS — WHAT WE MEASURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CONCORDANCE INDEX (CI / Harrell's C-index):
    CI = P(pred_i > pred_j | label_i > label_j)
    i.e., the fraction of all orderable pairs where the model correctly
    identifies which drug has higher affinity.

    CI = 0.5 → random guessing
    CI = 1.0 → perfect ranking
    CI = 0.0 → perfectly inverted (adversarially bad)

    Interpretation:
      CI ≈ 0.5: cold start (no context), model can't distinguish drugs
      CI ≈ 0.6–0.7: some context, noisy signal
      CI ≈ 0.75–0.88: warm episodes with good context

ENRICHMENT FACTOR AT K% (EF@k):
    EF@5% = (fraction of top binders in top 5% of ranked list) /
            (fraction of top binders by random chance)
    > 1.0 means better than random; EF@5% = 5 means 5× better than random.
    This directly measures hit-rate for virtual screening.

REGIME CLASSIFICATION:
    Cold: n_ctx = 0 (no context available)
    Cool: n_ctx = 1–4
    Warm: n_ctx = 5–19
    Hot:  n_ctx ≥ 20

    We report average CI per regime. As the stream progresses, more episodes
    become warm/hot, and the overall performance improves. This breakdown
    diagnoses whether improvements come from better cold-start handling or
    better warm performance.

PART 14: THE FULL FORWARD PASS — STEP BY STEP
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

For one episode (Protein A, query drugs D₁...Dₙ):

STEP 1 — Compute prior:
    prior[i] = gnn_prior.predict(A, D_i) for each query drug D_i
    This is a fast lookup into precomputed protein/drug embeddings +
    BilinearScoringHead dot product + label_offset.

STEP 2 — Compute residuals in history:
    For each revealed (X, Y, label) in history:
        residual = label - prior(X, Y)
    Context stores residuals so the GP only models unexplained deviation.

STEP 3 — Build context via product kernel:
    For each query drug D_i:
        score each history entry (X_j, Y_j) by k_prot(A,X_j) × k_drug(D_i,Y_j)
        take top K = 32 entries
        ctx_proteins[i, :K] = protein features of top entries
        ctx_drugs[i, :K] = drug features of top entries (Y_j, not D_i!)
        ctx_affinities[i, :K] = residuals of top entries
        ctx_mask[i, :K] = True

STEP 4 — Encode query pairs (deep kernel):
    Q[i] = DrugConditionalEncoder(protein_feat[A], drug_feat[D_i])   [d]

STEP 5 — Encode context pairs (cross-pair kernel):
    K_emb[i, j] = DrugConditionalEncoder(ctx_proteins[i,j], ctx_drugs[i,j])  [d]
    Note: ctx_drugs[i,j] is Y_j (observation drug), not D_i (query drug).
    This is the cross-pair kernel: enc(A,D_i)^T enc(X_j,Y_j) / sqrt(d).

STEP 6 — GP transfer (single-head attention over scalar residuals):
    transfer_logits[i, j] = Q[i] · K_emb[i,j]^T / sqrt(d)
    transfer_weights[i, :] = softmax(transfer_logits[i, :]) masked by ctx_mask
    transfer[i] = Σ_j transfer_weights[i,j] × ctx_affinities[i,j]

STEP 7 — Correction (multi-head cross-attention over d-dim context):
    x = Q.unsqueeze(1)                        [n_qry, 1, d]
    for layer in CrossAttentionLayer × 2:
        x = layer(query=x, context=K_emb, mask=ctx_mask)
    attended = LayerNorm(x).squeeze(1)        [n_qry, d]
    output = MLP(concat(Q, attended))         [n_qry, 2]
    mu_residual = output[:, 0]
    log_sigma = output[:, 1]

STEP 8 — Context gate:
    alpha = sigmoid(linear(attended))         [n_qry]
    mu_ctx = prior + alpha × (transfer + mu_residual)
    mu = mu_ctx where has_context else prior
    sigma = softplus(log_sigma) + ε

STEP 9 — Loss:
    TNPLoss(mu, sigma, labels) → NLL + ListNet + LambdaMART weighted sum
    Backward. Gradient step on GPAffinityModel parameters (φ, attention layers,
    correction head, ctx_gate, TNPLoss.log_vars).
    Prior (GNNPrior / MolGraphPrior) is FROZEN — not updated during streaming.

PART 15: WHAT EACH COMPONENT LEARNS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

MolGraphPrior (frozen after pretrain):
    Learns to predict absolute affinities from 3D structure. Provides μ₀.
    Also provides the drug embedding space used in product-kernel retrieval.

DrugConditionalEncoder φ (trained online during streaming):
    Learns the cross-pair kernel geometry. Answers: "for this drug, which
    structural features of proteins determine binding?" Geometry is specific
    to the drug class being screened. Evolves as the stream progresses.

CrossAttentionLayer (trained online):
    Learns to attend to the most relevant context observations. Multiple heads
    capture different aspects of binding similarity. Residual connections
    prevent attention from collapsing.

Correction head (trained online):
    Learns systematic nonlinearities that the GP transfer misses. Zero-init
    means it starts neutral and only learns corrections supported by data.

Context gate α (trained online):
    Learns when to trust the GP vs. the prior. With noisy or irrelevant context
    (poor drug-protein matches in the pool), α stays low and predictions stay
    anchored to the informative prior.

TNPLoss.log_vars (trained online):
    Learns the relative reliability of NLL vs. ranking losses. If ranking
    losses are noisy for a particular distribution, they get downweighted.

PART 16: SUMMARY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The model is a learned Gaussian Process for drug-protein binding affinity:

  Prior:          MolGraphPrior — physics-informed baseline from 3D structure
  Kernel:         DrugConditionalEncoder — learned cross-pair deep kernel
  Context:        Product-kernel retrieval from accumulated binding history
  Posterior mean: Approximated by single-head attention over scalar residuals
  Correction:     Nonlinear residual from multi-head cross-attention (zero-init)
  Uncertainty:    Calibrated sigma from correction head + NLL training
  Gate:           Learned α balancing GP vs. prior based on context quality
  Loss:           NLL + ListNet + LambdaMART with learned task weights
  Protocol:       Prequential — strictly test-then-train, no label leakage

The math is principled: GP regression is the optimal Bayesian estimator for
function interpolation with a given kernel. The model instantiates this
framework with a learned kernel (deep kernel learning) and approximates the
O(n³) posterior with O(n) attention. Every architectural choice maps directly
to a GP concept — this is not a "GP-inspired" model with ad hoc additions.
It IS a GP, with neural networks replacing the hand-crafted components.

As the streaming experiment runs, n_ctx grows, context quality improves, and
the kernel φ adapts to the binding landscape. Cold episodes (n_ctx = 0) fall
back to the prior honestly. Warm episodes (n_ctx ≥ 10) achieve CI 0.75–0.88.
The streaming improvement IS the GP posterior convergence.
